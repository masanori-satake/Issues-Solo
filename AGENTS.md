# Issues-Solo: AIエージェントおよび開発者ガイド (AGENTS.md)

このドキュメントは、Issues-Solo の設計思想、過去の重要な判断の背景（Why）、および AI エージェント（あなた）や開発者が開発・保守を行う際の具体的な行動指針をまとめたものです。

## 1. はじめに
本プロジェクトは Public なオープンソース・プロジェクトとして公開されており、不特定多数のユーザーが参照することを前提としています。
ユーザーの認知負荷を最小化し、データの透明性と安全性を最大限に高めることを最優先としています。

### Issues-Solo の位置づけ
Issues-Solo は JIRA 利用者のための「作業コンテキスト管理ツール」です。
- **狙い:** 「どのタブで何をしていたか」をサイドパネルから一目で把握し、作業の切り替えコストを最小化すること。
- **非目標:** 高度なタスク管理、チーム共有、外部サービスとの連携。

## 2. 設計思想の要約
詳細な設計思想、技術選定、および判断の背景については、製品仕様書 [docs/spec.md](docs/spec.md) に集約されています。開発時は常に `spec.md` を最新の正解として参照してください。

### コア・フィロソフィーの要点
- **持続可能性 (Sustainability):** 外部サーバーに依存せず、ブラウザ内（オフライン）で完結すること。
- **ミニマリズム (Minimalism):** 必要十分な機能に絞り、情報のノイズを最小化すること。
- **透明性 (Transparency):** データの保存先や通信の遮断を明示し、ユーザーが安心して利用できること。
- **保守性・長寿命 (Maintainability & Longevity):** 外部フレームワークを一切使用せず、標準的な Web API (Vanilla JS) のみで構築すること。

## 3. 開発・行動ルール

AI エージェントおよび開発者は、以下のルールを厳格に遵守しなければなりません。

### 3.1. 深い計画プロトコル (Deep Planning Protocol) - 対話の原則
実行前に「絶対的な確信」を得るための対話を行う。
- 1. 当たり前に思えることでも、すべての前提条件を確認するために質問を投げ、要件を完全に把握する。
- 2. コードから推測可能な詳細ではなく、ユーザーの意図や「何を実現したいか（Desire）」に焦点を当てて質問する。
- 3. ユーザーの回答を咀嚼し、不明点がゼロになるまで深掘りした質問を継続する。
- 4. すべてのコミュニケーションは `request_user_input` または `message_user` を通じて行い、独断を避ける。
- 5. 確信を得た後、`set_plan` で計画を立て、承認を得てから自律的な実行へ移る。承認後の追加質問は最小限に留める。

### 3.2. 言語統制 (Language Control) - 最優先事項
あらゆる外部コミュニケーションにおいて**日本語を最優先で使用し、英語の使用を最小限に留めなければならない**。

以下のすべての場面において、日本語を使用せよ：
- **対話:** `request_user_input` や `message_user` によるチャット。
- **PR作成:** タイトル（`title`）および説明文（`description`）。
- **コミットメッセージ:** Git のコミットメッセージ（`feat: ` などのキーワード以降）。
- **コメント返信:** レビューに対する回答。

### 3.3. 設計方針とデザイン
- **Google Material 3 (M3) 準拠:** デザインシステムとして M3 を採用する。詳細は [docs/DESIGN.md](docs/DESIGN.md) を参照。
- **Solo シリーズとの親和性:** QuickLog-Solo, Replace-Solo とデザイン言語・操作感を統一する。
- **技術スタック制限:** 外部OSSライブラリ（UIフレームワーク、ユーティリティ等）の使用を厳禁し、純粋な HTML/CSS/Vanilla JS のみで実装する。

### 3.4. 実装・保守の原則
- **仕様への忠実性:** すべての変更は `spec.md` の設計思想に合致しなければならない。
- **ルートディレクトリのクリーンネス:** 一時的なスクリプトをコミットしてはならない。
- **innerHTML の使用禁止:** `textContent` または DOM API (createElement等) を使用せよ。
- **Why を記録せよ:** 重要な変更を行う際は、単に「何を変えたか」だけでなく、「なぜその方法を選んだか」を `spec.md` または `AGENTS.md` に追記せよ。
- **コード保守性:** デッドコード、未使用変数、冗長なロジックを排除し、KISSの原則を徹底せよ。

## 4. 設計原則の適用 (Design Principles)
- **KISS (Keep It Simple, Stupid):** ロジックはシンプルに保つ。
- **DRY (Don't Repeat Yourself):** ロジックの重複を避ける。
- **SLAP (Single Level of Abstraction Principle):** 抽象度をそろえる。
- **OCP (Open-Closed Principle):** 拡張に対して開き、修正に対して閉じる。
- **YAGNI (You Ain't Gonna Need It):** 今必要ないものは作らない。
